## 二、服务器规格选型

### ⚠️ 抢占式实例的特殊处理：固定内网 IP 方案

**问题根源**：阿里云抢占式实例（Preemptible Instance）在被释放后重建，主网卡的私有 IP 会发生变化，导致 Kafka、Flink、Doris 等集群组件的配置全部失效。

**解决方案：弹性网卡（ENI）+ 固定私有 IP**

核心思路是将每台机器的"服务 IP"与"实例"解耦：
1. 为每台节点额外购买一块**辅助弹性网卡（Secondary ENI）**，绑定固定私有 IP
2. 所有中间件的配置文件只使用这个 ENI 的 IP，不使用主网卡 IP
3. 实例被释放后，ENI 和它绑定的固定 IP 不受影响
4. 购买新的抢占式实例后，将 ENI 重新挂载到新实例上，服务恢复，IP 不变

**操作步骤（阿里云控制台，购买 ECS 之后执行）：**

```
控制台路径：云服务器 ECS → 网络与安全 → 弹性网卡

1. 创建 3 块 ENI，与 ECS 处于同一 VPC + 交换机（vSwitch）
2. 为每块 ENI 指定固定私有 IP：
   - master-01 的 ENI：192.168.1.10
   - worker-01 的 ENI：192.168.1.20
   - worker-02 的 ENI：192.168.1.30
3. 将每块 ENI 分别挂载到对应实例（实例详情页 → 网络 → 绑定弹性网卡）
4. 在每台实例的 OS 内启用 ENI（Ubuntu 需要配置 netplan）
5. 后续所有软件配置全部使用上述 192.168.1.x 地址
```

**Ubuntu 内启用 ENI（每台实例执行）：**

```bash
# 查看网卡设备名（通常主网卡 eth0，ENI 是 eth1）
ip link show

# 编辑 /etc/netplan/50-cloud-init.yaml，追加 eth1 配置
sudo tee -a /etc/netplan/50-cloud-init.yaml << 'EOF'
        eth1:
            addresses:
            - 192.168.1.10/24   # 根据节点替换为对应 IP
            dhcp4: false
EOF

sudo netplan apply
# 验证
ip addr show eth1
```

**实例被释放后的恢复流程（< 10 分钟恢复）：**

```
1. 购买同规格新的抢占式实例（同 VPC + 交换机）
2. 控制台：将旧实例的 ENI 解绑，挂载到新实例
3. 新实例内执行 netplan 配置（可写成 cloud-init 脚本自动化）
4. 所有软件配置无需修改，直接启动服务
```

### 节点角色规划

| 节点 | ENI 固定 IP | 角色 | 推荐实例规格 | vCPU | 内存 | 系统盘 |
|---|---|---|---|---|---|---|
| `master-01` | 192.168.1.10 | Kafka / Flink JobManager / MySQL / Redis / Doris FE / Spring Boot / AI Engine / Prometheus / Grafana | **ecs.g7.2xlarge**（或同等抢占式） | 8 核 | 32 GB | **80 GB ESSD PL1** |
| `worker-01` | 192.168.1.20 | Flink TaskManager / Doris BE | **ecs.g7.2xlarge** | 8 核 | 32 GB | **80 GB ESSD PL1** |
| `worker-02` | 192.168.1.30 | Flink TaskManager / Doris BE / Milvus | **ecs.g7.2xlarge** | 8 核 | 32 GB | **80 GB ESSD PL1** |

> **为什么建议 32GB 而不是 16GB**：Doris BE 建议最低 16GB，加上 Flink TaskManager（8GB）、RocksDB（2GB）、OS（4GB），16GB 节点在正常负载下就会 OOM。32GB 是流畅运行的保障线，抢占式实例 32GB 费用极低（通常比 16GB 按量实例便宜 70%+）。

### 附加云服务

| 服务 | 用途 | 说明 |
|---|---|---|
| **阿里云 OSS（日本东京）** | Paimon 数据湖底层存储 + Flink Checkpoint | 替代 HDFS，按量计费，无需运维 |
| **弹性公网 IP（EIP）** | master-01 对外访问入口 | EIP 可随时绑定到新实例，公网 IP 不变 |
| **弹性网卡（ENI × 3）** | 固定集群内网 IP | 每块 ENI 费用极低，解决抢占式 IP 漂移问题 |

## 三、操作系统与基础软件环境

### 操作系统

**推荐：Ubuntu 22.04.3 LTS（64-bit x86）**

理由：大数据组件官方文档以 Ubuntu 为基准，`apt` 体验好，LTS 支持至 2027 年。

### 所有节点基础初始化脚本

```bash
# 1. 系统更新
sudo apt-get update && sudo apt-get upgrade -y

# 2. 必备工具
sudo apt-get install -y \
  curl wget git vim htop \
  net-tools lsof unzip \
  build-essential \
  python3.12 python3.12-venv python3-pip  # 采集器 + AI Engine 语言

# 3. 时钟同步（集群必须时区一致，否则 Flink 事件时间窗口乱）
sudo timedatectl set-timezone Asia/Tokyo
sudo apt-get install -y ntpdate
sudo ntpdate -u ntp.aliyun.com
# 写入 cron，每小时同步一次（抢占式实例重启后时钟可能飘）
(crontab -l 2>/dev/null; echo "0 * * * * /usr/sbin/ntpdate -u ntp.aliyun.com") | crontab -

# 4. 关闭 swap（Flink + Doris 强烈要求，swap 会导致性能抖动）
sudo swapoff -a
sudo sed -i '/ swap / s/^/#/' /etc/fstab

# 5. 调大文件描述符（Kafka 需要大量 FD）
echo "* soft nofile 65536" | sudo tee -a /etc/security/limits.conf
echo "* hard nofile 65536" | sudo tee -a /etc/security/limits.conf

# 6. 调整内核参数（Kafka + Flink 网络优化）
sudo tee -a /etc/sysctl.conf << 'EOF'
net.core.somaxconn=65535
net.ipv4.tcp_max_syn_backlog=65535
vm.swappiness=0
EOF
sudo sysctl -p
```

### /etc/hosts 配置（三台节点均需配置，使用 ENI 固定 IP）

```
192.168.1.10  master-01
192.168.1.20  worker-01
192.168.1.30  worker-02
```

### JDK 21 安装（三台节点，通过 SDKMAN 管理）

```bash
# 安装 SDKMAN
curl -s "https://get.sdkman.io" | bash
source "$HOME/.sdkman/bin/sdkman-init.sh"

# 安装 Eclipse Temurin JDK 21（OpenJDK 最佳发行版，Adoptium 出品）
sdk install java 21.0.3-tem

# 验证
java -version
# 期望输出：openjdk version "21.0.3" 2024-04-16 LTS
```

### Docker 安装（三台节点）

```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker ubuntu
# 重新登录后生效
docker --version
```

## Phase 0：基础设施搭建（预计 3 天）

### 【思路说明】

本阶段是所有后续工作的地基。最关键的事情有两件：
1. **ENI 固定 IP**：必须在安装任何软件前完成，后续所有配置文件都基于这个固定 IP，绝对不能用动态分配的主网卡 IP。
2. **节点互信（SSH 免密）**：后续 Flink 会从 JobManager SSH 到 TaskManager 分发任务，Flink standalone 脚本也依赖此。

本阶段完成的标志：从 master-01 执行 `ssh ubuntu@worker-01` 和 `ssh ubuntu@worker-02` 无需密码，且 `ping worker-01` 能通（通过 /etc/hosts 解析到 ENI 固定 IP）。

### Step 0-1：购买 ECS + OSS + ENI

**阿里云控制台操作（日本东京地域）：**

1. 购买 3 台抢占式实例
   - 规格：`ecs.g7.2xlarge`（8vCPU / 32GB）
   - 计费方式：**抢占式实例**，出价策略选"自动出价"（最省心）
   - 镜像：**Ubuntu 22.04 LTS 64位**
   - 系统盘：**80 GB ESSD PL1**（无数据盘）
   - 网络：新建 VPC `streamlake-vpc`，网段 `192.168.0.0/16`；新建交换机 `streamlake-vsw`，网段 `192.168.1.0/24`
   - 安全组 `streamlake-sg`：见第六节端口规划
   - 登录方式：**密钥对**（不用密码）

2. 创建 OSS Bucket
   - 名称：`streamlake-paimon-tokyo`（全球唯一，如名字被占用加后缀）
   - 地域：**日本东京（ap-northeast-1）**
   - 存储类型：标准
   - 权限：私有
   - 记录 **AccessKey ID / Secret**（子账号 RAM 权限：OSS 读写 + 该 Bucket）

3. 购买 3 块弹性网卡（ENI）
   - 控制台：网络与安全 → 弹性网卡 → 创建弹性网卡
   - 选择同一 VPC + 交换机
   - 分别指定私有 IP：`192.168.1.10`、`192.168.1.20`、`192.168.1.30`
   - 创建后，分别挂载到 master-01、worker-01、worker-02

4. 为 master-01 绑定 EIP（弹性公网 IP）

### Step 0-2：OS 内启用 ENI（三台节点）

```bash
# 登录实例后，查看 ENI 设备名
ip link show
# 通常主网卡是 eth0，ENI 是 eth1

# 找到 netplan 配置文件
ls /etc/netplan/
# 通常是 50-cloud-init.yaml

# 以 master-01 为例，追加 eth1 配置
# ⚠️ YAML 缩进严格，使用空格不用 Tab
sudo tee /etc/netplan/51-eni.yaml << 'EOF'
network:
  version: 2
  ethernets:
    eth1:
      addresses:
        - 192.168.1.10/24    # worker-01 改为 .20，worker-02 改为 .30
      dhcp4: false
EOF

sudo netplan apply
# 验证 ENI IP 已生效
ip addr show eth1 | grep "192.168.1"
```

### Step 0-3：配置 /etc/hosts（三台节点）

```bash
# 三台节点都执行
sudo tee -a /etc/hosts << 'EOF'
192.168.1.10  master-01
192.168.1.20  worker-01
192.168.1.30  worker-02
EOF

# 验证
ping -c 2 master-01
```

### Step 0-4：SSH 免密配置（master-01 → worker-01/02）

```bash
# 在 master-01 上生成密钥对
ssh-keygen -t ed25519 -f ~/.ssh/streamlake_cluster -N ""

# 将公钥分发到 worker 节点（需要先输一次密码）
ssh-copy-id -i ~/.ssh/streamlake_cluster.pub ubuntu@worker-01
ssh-copy-id -i ~/.ssh/streamlake_cluster.pub ubuntu@worker-02

# 配置 master-01 的 ~/.ssh/config
cat >> ~/.ssh/config << 'EOF'
Host worker-01
```

## Phase 2：数据采集层 Python asyncio Collector（预计 4 天）

### 【思路说明】

**为什么选 asyncio + websockets，不用 websocket-client**：
`websocket-client` 是同步阻塞库，每收到一条消息就会阻塞等待处理，无法同时处理多个连接。`websockets` 库基于 asyncio，用单线程事件循环并发处理所有连接，适合 I/O 密集型场景（网络收发）。这是两者的根本区别，面试时需要能解释。

**Binance WebSocket 的限制**：
- 单个 WS 连接最多订阅 **200 个 stream**
- 每个 stream 格式为 `{symbol}@trade` 或 `{symbol}@depth@100ms`
- 连接空闲 30 分钟会被服务器断开，需要定期发 `ping` 或自动重连
- 建议先订阅 Top-50 交易对，后期可扩展

**aiokafka vs confluent-kafka**：
- `aiokafka` 是纯 Python 异步实现，与 asyncio 无缝集成，代码简洁
- `confluent-kafka` 性能更高但是同步 API，需要在 asyncio 中用 `loop.run_in_executor` 包装，增加复杂度
- 本项目选 `aiokafka`，吞吐量完全够用（Top-50 交易对）

**消息格式**：直接推送 JSON 字符串到 Kafka，不做 Avro 序列化（Avro 需要 Schema Registry，增加复杂度，对简历价值提升有限）。Flink 消费时用 JSON 反序列化即可。

**验收标准**：
- 采集器运行 10 分钟无崩溃
- `kafka-console-consumer` 订阅 `binance.trade.raw`，能看到持续的 JSON 消息流
- 消息中包含字段：`symbol`, `price`, `qty`, `quoteQty`, `time`, `isBuyerMaker`

**binance/client.py**（带指数退避重连）：

```python
import asyncio
import json
import logging
import websockets
from websockets.exceptions import ConnectionClosed

logger = logging.getLogger(__name__)

class BinanceWSClient:
    def __init__(self, url: str, on_message, reconnect_base=5, reconnect_max=60):
        self.url = url
        self.on_message = on_message   # 回调函数（协程）
        self.reconnect_base = reconnect_base
        self.reconnect_max = reconnect_max
        self._running = False

    async def start(self):
        self._running = True
        delay = self.reconnect_base
        while self._running:
            try:
                logger.info(f"Connecting to {self.url}")
                async with websockets.connect(
                    self.url,
                    ping_interval=20,
                    ping_timeout=10,
                    max_size=2**22,    # 4MB，防止深度快照消息超限
                ) as ws:
                    delay = self.reconnect_base  # 连接成功，重置退避
                    logger.info("Connected.")
                    async for raw_msg in ws:
                        data = json.loads(raw_msg)
                        await self.on_message(data)

            except ConnectionClosed as e:
                logger.warning(f"WS closed: {e}. Reconnecting in {delay}s...")
            except Exception as e:
                logger.error(f"WS error: {e}. Reconnecting in {delay}s...")

            if self._running:
                await asyncio.sleep(delay)
                delay = min(delay * 2, self.reconnect_max)  # 指数退避

    def stop(self):
        self._running = False
```

**main.py**（asyncio 主入口）：

```python
import asyncio
import json
import logging
import yaml
from binance.client import BinanceWSClient
from binance.models import TradeEvent
from sink.kafka_sink import KafkaSink

logging.basicConfig(level=logging.INFO,
                    format='%(asctime)s %(levelname)s %(name)s: %(message)s')
logger = logging.getLogger("main")

async def main():
    with open("config.yaml") as f:
        cfg = yaml.safe_load(f)

    kafka_sink = KafkaSink(cfg["kafka"])
    await kafka_sink.start()

    symbols = cfg["binance"]["symbols"]
    # 构建 combined stream URL（单连接最多 200 个 stream）
    streams = []
    for sym in symbols:
        streams.append(f"{sym.lower()}@trade")

    # 每 200 个 stream 一个连接
    chunk_size = cfg["binance"]["max_streams_per_conn"]
    tasks = []
    for i in range(0, len(streams), chunk_size):
        chunk = streams[i:i+chunk_size]
        params = "/".join(chunk)
        url = f"{cfg['binance']['ws_base_url']}?streams={params}"

        async def on_message(data, sink=kafka_sink):
            if "stream" not in data:
                return
            stream_name = data["stream"]
            if "@trade" in stream_name:
                event = TradeEvent.from_raw(data)
                msg = event.model_dump_json().encode()
                await sink.send(cfg["kafka"]["topic_trade"], msg)

        client = BinanceWSClient(
            url=url,
            on_message=on_message,
            reconnect_base=cfg["binance"]["reconnect_delay_base"],
            reconnect_max=cfg["binance"]["reconnect_delay_max"],
        )
        tasks.append(asyncio.create_task(client.start()))
        logger.info(f"Started WS connection for {len(chunk)} streams")

    try:
        await asyncio.gather(*tasks)
    except KeyboardInterrupt:
        logger.info("Shutting down...")
    finally:
        await kafka_sink.stop()

if __name__ == "__main__":
    asyncio.run(main())
```

## Phase 3：流计算层 Flink 2.0（预计 10 天）

### 【思路说明】

**Flink 在架构中的核心地位**：Flink 是整个项目最复杂、技术含量最高的部分，也是简历的核心卖点。本项目写了三个作业，代表了 Flink 最有价值的三类使用场景：
- **Job-1 K 线聚合**：窗口聚合，展示 Window API + 事件时间处理 + Watermark 的使用
- **Job-2 巨鲸 CEP**：复杂事件处理，展示 CEP Pattern API 和 RocksDB 状态管理
- **Job-3 风控广播**：动态规则分发，展示 Broadcast State + Flink CDC 的结合

**RocksDB 状态后端选择理由**：
- Flink 默认的 HashMapStateBackend 把状态存 JVM 堆内存，状态大时容易 Full GC 甚至 OOM
- RocksDB 把状态存磁盘（/opt/flink/state），只把热数据缓存在内存，状态大小不受堆内存限制
- 对本项目：CEP 作业需要记录每个交易对的历史 tick 序列，可能有几十 MB，必须用 RocksDB

**Checkpoint 策略**：
- 每 30 秒做一次增量 Checkpoint，存到 OSS（不占本地磁盘）
- 增量 Checkpoint：只上传自上次 CP 以来变化的 RocksDB SST 文件，速度快
- 保留最近 3 个 Checkpoint，防止 OSS 存储膨胀

**三个作业独立部署，互不影响**：每个作业是独立的 JAR，独立提交，独立监控，一个作业崩溃不影响另外两个。

**验收标准**：
- Flink Web UI（`http://master-01:8081`）能看到三个 Running Job
- Doris 中 `kline_1min` 表有持续新增数据
- Doris 中 `whale_alert` 表在有大单时有记录
- Redis 中 `risk:blacklist:*` 有 Key（需要触发风控规则）

### Job-1：K 线聚合（核心逻辑说明）

```
数据流：
Kafka(binance.trade.raw)
  → TradeEventDeserializer（JSON → TradeEvent POJO）
  → assignTimestampsAndWatermarks（事件时间 + 允许 5s 乱序）
  → keyBy(symbol)
  → TumblingEventTimeWindow(1 分钟)
  → OHLCVAggregateFunction
       open  = 第一条 price
       high  = max(price)
       low   = min(price)
       close = 最后一条 price
       volume = sum(qty)
       quoteVolume = sum(price * qty)
       tradeCount = count()
  → Sink-1: PaimonSink（历史明细，写 OSS，append-only）
  → Sink-2: DorisSink（实时 K 线内表，UNIQUE KEY，支持覆盖写）
```

关键技术点：
- **Watermark 策略**：允许 5 秒乱序（网络延迟导致消息乱序），超过 5 秒的迟到数据写入 side output 单独处理
- **Doris Sink**：使用 Flink Doris Connector，StreamLoad 方式写入，Exactly-Once 语义通过两阶段提交保障

### Job-2：巨鲸 CEP（核心逻辑说明）

```
检测模式：60 秒内，同一交易对同方向（买/卖）累计成交额超过阈值
阈值：BTC=50万 USDT，主流币=20万 USDT，山寨币=5万 USDT（从 MySQL 读取，可配置）

Pattern 设计：
Pattern.<TradeEvent>begin("bigTrade")
    .where(trade -> trade.quoteQty > singleTradeThreshold)  // 单笔大单
    .timesOrMore(3)                                          // 连续出现 3 次以上
    .greedy()
    .within(Time.seconds(60))                                // 60 秒窗口

⚠️ CEP 状态 TTL 设置（防止状态爆炸）：
StateTtlConfig ttl = StateTtlConfig
    .newBuilder(Time.minutes(5))    // 5 分钟未触发则清理状态
    .setUpdateType(UpdateType.OnCreateAndWrite)
    .build();
```

### Job-3：风控广播（核心逻辑说明）

```
两路流 Join：
流-A（控制流）：MySQL risk_rules 表 → Flink CDC Source → BroadcastStream
流-B（数据流）：Kafka binance.trade.raw → TradeEvent

处理逻辑（RiskControlProcessFunction）：

- Doris risk_trigger 表 INSERT              → 记录触发日志
```

## Phase 4：湖仓存储层 Paimon + OSS + Doris（预计 7 天）

### 【思路说明】

**湖仓一体（Lakehouse）的核心理念**：传统架构中，数据仓库（Warehouse）存热数据、查询快但贵；数据湖（Lake）存冷数据、便宜但查询慢。Lakehouse 把两者结合：用 Paimon（湖格式）存原始明细（便宜），用 Doris（OLAP 引擎）存聚合结果（快速查询），Doris 通过 External Catalog 也能直接查 Paimon 数据，做联邦查询。

**为什么用 OSS 而不是 HDFS**：
- HDFS NameNode + DataNode 需要额外的进程和内存，在 80GB 磁盘场景下太重
- OSS 是阿里云托管的对象存储，Paimon 1.1 原生支持 OSS 协议
- 不需要运维，数据高可用，按量付费
- 简历可写"基于对象存储（OSS）的云原生湖仓"，这是更现代的架构

**Paimon 的 Compaction 风险**：
- Paimon 底层是 LSM-Tree，高频写入会积累 sorted run，达到阈值触发 compaction
- Compaction 期间 CPU 飙升、读延迟增大
- 缓解方案：把 compaction 作为独立 Flink job 运行，与写入作业隔离

**Doris 表类型选择**：
- 实时 K 线（需要覆盖更新）→ **UNIQUE KEY 表**（Merge-On-Write 模式）
- 巨鲸告警（只追加）→ **DUPLICATE KEY 表**
- 不要用 AGGREGATE KEY，它在查询时才合并，读性能差

**验收标准**：
- `SELECT COUNT(*) FROM doris.kline_1min` 每分钟增加约 50 条（Top-50 交易对）
- Doris `External Catalog` 联邦查询 Paimon 能返回原始 tick 数据
- Paimon OSS 目录下能看到 parquet 文件生成

## Phase 5：数据服务与 AI 层（预计 7 天）

### 【思路说明】

**Spring Boot 作为 API 网关的角色**：
- 对前端提供 REST API（历史 K 线查询、告警列表）
- 对前端提供 WebSocket 端点（实时推送新 K 线、鲸鱼告警）
- 向 Python AI Engine 发起 HTTP 调用（RAG 诊断）

**Virtual Threads（JDK 21 核心特性）**：
- 传统 Spring MVC：一个 HTTP 请求占用一个平台线程，默认 200 个线程 → 最多 200 并发
- Virtual Threads：轻量级线程，IO 等待时自动挂起不占 OS 线程，百万并发不是梦
- 开启方式：`spring.threads.virtual.enabled=true`，只需一行配置，代码无需改动
- 面试时能讲清楚"为什么不用 WebFlux"："Virtual Threads 代码是同步风格但效果接近响应式，可读性更好"

**RAG（检索增强生成）的工作流**：
这是项目的 AI 亮点，需要理解完整流程：
1. **离线建库**：定时把历史鲸鱼告警、异常 K 线事件 Embedding，存入 Milvus 向量库
2. **在线检索**：用户请求诊断时，把"当前异动描述"向量化，从 Milvus 检索最相似的历史案例 Top-5
3. **生成报告**：把"当前指标 + 历史案例 + Prompt 模板"发给 LLM，得到 Markdown 诊断报告
4. **存储报告**：写入 Doris `ai_diagnosis` 表，供下次查询

**验收标准**：
- `GET /api/kline/BTCUSDT?interval=1min&limit=100` 返回 100 条 K 线 JSON
- WebSocket 连接后，每分钟收到新 K 线推送
- `POST /api/ai/diagnose {"symbol":"BTCUSDT"}` 返回 Markdown 报告

## Phase 6：前端实时大屏（预计 7 天）

### 【思路说明】

**TradingView Lightweight Charts 的重要性**：
这个库是 TradingView 官方开源的，Binance 等专业交易所的 K 线图都基于此实现。相比 ECharts/G2Plot，它专门为金融时序数据优化，性能极好（10万+ 数据点流畅渲染），且 API 设计适合实时更新（`series.update(bar)` 一行代码更新最新 K 线）。简历上用这个库，面试官一眼就知道你了解金融前端生态。

**WebSocket 实时推送设计**：
前端订阅 Spring Boot 的 WebSocket 端点，服务端在新 K 线形成（每分钟）时主动推送。这样避免了轮询（polling），减少服务器压力，且延迟更低。

**大屏布局设计原则**：
- 暗黑主题（金融大屏标配）：背景 `#0d1117`，强调色 `#00d09e`（涨）`#ef5350`（跌）
- 信息密度高但不拥挤：主图占 60% 宽度，右侧 40% 显示告警和概览

**验收标准**：
- 大屏打开后，K 线图实时更新，不需要手动刷新
- 有大单时，右侧告警面板出现新条目
- AI 诊断按钮点击后，悬浮抽屉显示 Markdown 报告（有 Loading 状态）

## Phase 7：可观测性 Prometheus + Grafana（预计 3 天）

### 【思路说明】

**为什么可观测性是简历的必需项**：
能部署 Prometheus + Grafana 并设置告警规则，说明你有生产级系统思维，不只会写功能代码。面试时能展示监控大屏（"你看，Kafka 消费延迟在这里，Flink Checkpoint 耗时在这里，内存使用在这里"），会让面试官印象深刻。

**监控的四个黄金信号（Four Golden Signals）**：
- **延迟（Latency）**：Kafka 消费积压、Flink Checkpoint 耗时、API P99 响应
- **流量（Traffic）**：每秒处理消息数、API QPS
- **错误（Errors）**：Flink 作业失败次数、WS 断连次数、API 5xx 比率
- **饱和度（Saturation）**：CPU 使用率、内存使用率、磁盘使用率

**验收标准**：
- Grafana `http://master-01:3000` 能打开，有仪表盘
- 能看到 Flink 作业的 Checkpoint 耗时曲线
- Kafka 消费积压（Lag）指标实时可见
